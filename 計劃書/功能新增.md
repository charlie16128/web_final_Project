# 功能新增.md

> 目的：將 GitHub issues 中想新增的功能整理成可交給 Codex 實作的階段式開發文件。  
> 原則：不要一次大改；同一頁面、同一區塊、會互相影響位置的功能要一起做。每個階段完成後再進入下一階段，不能讓現有的功能消失

---

## 一、目前確認的設計規則

### 1. 收藏功能
- 收藏的專案不要另外做新頁面。
- 主畫面「搜尋專題」的篩選下拉選單新增一個選項：`已收藏`。
- 使用者選擇 `已收藏` 時，只顯示自己收藏過的專案。
- 專案卡片上要有收藏按鈕。
- 收藏狀態要能即時切換。

### 2. 通知中心
- 通知按鈕放在右上角。
- 點擊後在右上角展開一個懸浮小視窗。
- 不跳新頁面。
- 懸浮視窗要覆蓋在原本畫面上方，不影響原本排版。
- 通知內容包含：
  - 申請加入專案
  - 申請結果
  - 留言被回覆
  - 專案內容被修改
  - 被移出隊伍
  - 檢舉處理結果
  - 管理員相關通知

### 3. 專案詳情 / 群組討論區版面
在群組討論上方新增兩個區塊，順序固定：

1. 公告區
2. 日期倒數區
3. 群組聊天 / 討論區

規則：
- 公告區只顯示一行摘要。
- 日期倒數區只顯示一行摘要。
- 點公告區的 div 可以打開公告懸浮視窗。
- 點日期倒數區的 div 可以打開倒數懸浮視窗。
- 懸浮視窗顯示完整內容。
- 公告與倒數都不要讓卡片高度過高，避免壓縮聊天區。

### 4. 檢舉功能
需要有檢舉按鈕的位置：
- 專案
- 留言
- 評價留言

不需要有檢舉按鈕的位置：
- 個人頁面本身

個人頁面下方的「評價留言」需要檢舉按鈕。

### 5. 管理員帳號與權限
固定超級管理員帳號：

```txt
username: admin2006
email: admin@gmail.com
password: admin2006
```

管理員規則：
- `admin2006` 是唯一超級管理員。
- `admin2006` 可以到別人的個人頁面，把該使用者設定為管理員。
- 被設定成管理員的人可以處理管理員功能。
- 被設定成管理員的人不能再新增其他管理員。
- 所有管理員顯示名稱都要加上紅色 Prefix：`[ADMIN]`
- `[ADMIN]` 要出現在所有顯示使用者名稱的地方，包括：
  - 專案
  - 留言
  - 群組聊天
  - 評價留言
  - 通知
  - 個人頁面
  - 管理員頁面

---

## 二、開發階段總覽

不要一次全部改完，請依照以下階段實作。

| 階段 | 主要頁面 | 主要功能 | 原因 |
|---|---|---|---|
| 階段 1 | 資料庫 / 登入 / 使用者資料 | 管理員角色、admin2006、使用者資料欄位 | 其他功能都會依賴使用者權限 |
| 階段 2 | 主畫面 / 專案列表 | 收藏、已收藏篩選、技能篩選、專案人數狀態 | 都影響專案卡片與列表篩選 |
| 階段 3 | 專案詳情頁 | 公告、日期倒數、群組聊天上方版面 | 同一頁面的排版會互相影響 |
| 階段 4 | 專案詳情 / 隊伍管理 | 邀請成員、隊長轉移、滿員關閉申請 | 都跟隊伍狀態有關 |
| 階段 5 | 個人頁面 | 個人資料、技能、自我介紹、GitHub、合作評價 | 都在個人頁面一起排版 |
| 階段 6 | 通知中心 | 右上角通知懸浮視窗、通知資料 | 需要串接前面功能事件 |
| 階段 7 | 檢舉與管理員頁面 | 檢舉專案 / 留言 / 評價、管理員處理 | 需要使用者與通知功能完成後再做 |
| 階段 8 | 整合測試與 UI 修正 | 全站檢查、權限檢查、RWD | 避免前面分階段產生衝突 |

# 階段 1：使用者資料、登入、管理員角色

## 目標
先把權限基礎建立好，後面所有頁面才能判斷誰是管理員。

## 需要修改
### SQLite users table
請確認或新增以下欄位：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
username TEXT NOT NULL
email TEXT NOT NULL UNIQUE
password TEXT NOT NULL
role TEXT DEFAULT 'user'
avatar TEXT
department TEXT
grade TEXT
skills TEXT
bio TEXT
github_url TEXT
is_suspended INTEGER DEFAULT 0
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
```

### 初始化 admin2006
如果資料庫沒有此帳號，啟動 server 時自動建立，需要注意我有部署到azure平台上，sql會被重新建立，因此要寫死在db.js當建立時候就要有該帳號：

```txt
username: admin2006
email: admin@gmail.com
password: admin2006
role: super_admin
```

如果已有此 email，請確保：
- username 是 `admin2006`
- role 是 `super_admin`

### 權限規則
```txt
user: 一般使用者
admin: 管理員
super_admin: 超級管理員，只能是 admin2006
```

### 前端顯示名稱規則
建立共用函式或共用元件：

```js
formatDisplayName(user)
```

規則：
- 如果 `role === 'admin'` 或 `role === 'super_admin'`
  - 顯示紅色 `[ADMIN]`
  - 後面接使用者名稱
- 一般使用者只顯示名稱

範例：
```html
<span class="admin-prefix">[ADMIN]</span> admin2006
```

CSS：
```css
.admin-prefix {
  color: #e53935;
  font-weight: 700;
  margin-right: 4px;
}
```

## 階段 1 完成標準
- admin2006 可以登入。
- admin2006 的 role 是 `super_admin`。
- 一般使用者 role 是 `user`。
- 管理員名稱在所有已套用元件中可顯示紅色 `[ADMIN]`。

# 階段 2：主畫面專案列表、收藏、篩選、人數狀態

## 目標
集中修改主畫面與專案卡片，不要分散到其他頁面。

## 功能 1：收藏專案
### 資料表
新增 `project_favorites`：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
user_id INTEGER NOT NULL
project_id INTEGER NOT NULL
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
```

限制：
```sql
UNIQUE(user_id, project_id)
```

### API
```txt
POST /api/projects/:id/favorite
DELETE /api/projects/:id/favorite
GET /api/projects?filter=favorited
```

### 前端
專案卡片右上角新增收藏按鈕：
- 未收藏：空心星星或空心愛心
- 已收藏：實心星星或實心愛心

按下後：
- 不要跳頁
- 直接更新畫面狀態
- 未登入者點擊時提示需要登入

---

## 功能 2：搜尋專題下拉選單新增「已收藏」
原本主畫面的篩選下拉選單新增：

```txt
全部
已收藏
依技能 / 技能標籤
其他原本已有的分類
```

當選擇 `已收藏`：
- 只顯示目前登入使用者收藏的專案。
- 如果沒有收藏，顯示空狀態：
  - `目前沒有收藏的專案`

---

## 功能 3：技能標籤篩選
專案需要有 skills / tags 欄位。

### projects table 新增或確認
```sql
skills TEXT
```

可先用逗號字串儲存：
```txt
Vue,Node.js,SQLite
```

### 前端
- 專案卡片顯示技能標籤。
- 篩選區可以依技能顯示專案。
- 可與搜尋關鍵字一起作用。

---

## 功能 4：目前人數 / 缺額 / 滿員狀態
專案卡片顯示：

```txt
目前 2 / 4 人
缺 2 人
```

若滿員：
```txt
已滿員
```

滿員規則：
- `current_members >= max_members`
- 滿員後自動關閉申請按鈕。
- 按鈕顯示 `已滿員` 並 disabled。

## 階段 2 完成標準
- 主畫面可收藏 / 取消收藏。
- 篩選下拉選單有 `已收藏`。
- 技能篩選可用。
- 專案卡片有人數狀態。
- 滿員專案不能申請。

# 階段 3：專案詳情頁公告、日期倒數、群組聊天上方版面

## 目標
同一頁面的區塊位置一次整理，避免之後反覆調整排版。

## 版面順序
在專案詳情頁的群組討論區上方，固定順序：

```txt
[公告區 div]
[日期倒數 div]
[群組聊天 / 討論區]
```

---

## 功能 1：公告區
### 資料表
新增 `project_announcements`：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
project_id INTEGER NOT NULL
author_id INTEGER NOT NULL
content TEXT NOT NULL
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
updated_at DATETIME
```

### 顯示規則
公告區只顯示一行：

```txt
公告：今天晚上 8 點討論分工...
```

如果沒有公告：
```txt
目前沒有公告
```

### 互動
- 點整個公告 div 開啟懸浮視窗。
- 懸浮視窗顯示完整公告內容。
- 隊長或管理員可以新增 / 編輯 / 刪除公告。
- 一般成員只能查看。

---

## 功能 2：日期倒數
### 資料表
新增 `project_deadlines`：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
project_id INTEGER NOT NULL
title TEXT NOT NULL
deadline_date DATE NOT NULL
description TEXT
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
updated_at DATETIME
```

### 顯示規則
日期倒數區只顯示一行，例如：

```txt
期末報告倒數：剩 12 天
```

如果有多個倒數：
- div 只顯示最近的一個。
- 點 div 後在懸浮視窗看全部倒數。

如果沒有倒數：
```txt
目前沒有設定倒數日期
```

### 互動
- 點整個日期倒數 div 開啟懸浮視窗。
- 懸浮視窗顯示所有倒數日期。
- 隊長或管理員可以新增 / 編輯 / 刪除倒數。
- 一般成員只能查看。

---

## 功能 3：群組聊天區調整
聊天區維持在公告與倒數下方。

聊天訊息需要顯示：
- 使用者名稱
- 管理員 `[ADMIN]` prefix
- 訊息內容
- 發送時間
- 回覆按鈕
- 檢舉按鈕

## 階段 3 完成標準
- 公告、倒數、聊天順序正確。
- 公告和倒數都只佔一行。
- 點公告 div 可打開完整公告懸浮視窗。
- 點倒數 div 可打開完整倒數懸浮視窗。
- 不會因公告或倒數內容太長導致聊天區被擠壓。

# 階段 4：專案詳情與隊伍管理功能

## 目標
處理隊伍相關功能，因為邀請、轉移隊長、滿員都會互相影響。

## 功能 1：主動邀請成員
隊長可以在專案詳情頁邀請其他使用者加入。

### API
```txt
POST /api/projects/:id/invitations
GET /api/me/invitations
POST /api/invitations/:id/accept
POST /api/invitations/:id/reject
```

### 規則
- 只有隊長可以邀請。
- 專案滿員不能邀請。
- 被邀請者會收到通知。
- 接受邀請後加入成員。
- 拒絕邀請後通知隊長。

---

## 功能 2：隊長轉移
隊長可以把隊長身份轉移給同專案成員。

### 規則
- 只能轉移給目前專案成員。
- 轉移後原隊長變成一般成員。
- 新隊長獲得管理公告、倒數、成員、邀請的權限。
- 轉移成功後通知所有成員。

---

## 功能 3：滿員自動關閉申請
規則：
- 當目前人數等於最大人數，申請與邀請功能 disabled。
- 如果有人離開，重新開放申請。
- 滿員狀態要同步影響：
  - 主畫面專案卡片
  - 專案詳情頁
  - 邀請功能
  - 申請加入功能

## 階段 4 完成標準
- 隊長可以邀請成員。
- 被邀請者能接受 / 拒絕。
- 隊長能轉移給其他成員。
- 滿員時申請與邀請都不能使用。

---

# 階段 5：個人頁面與合作評價

## 目標
集中處理個人資料顯示與評價留言，避免跟管理員與檢舉功能分散修改。

## 功能 1：個人資料
個人頁面需要顯示與編輯：

```txt
個人照片
系級
年級
技能
自我介紹
GitHub 網址
```

### 欄位
使用 users table：

```sql
avatar TEXT
department TEXT
grade TEXT
skills TEXT
bio TEXT
github_url TEXT
```

### 規則
- 使用者可以編輯自己的資料。
- 其他人只能查看。
- 管理員可以查看。
- `super_admin` 可以在別人的個人頁面設定對方為管理員。

---

## 功能 2：合作評價
### 資料表
新增 `user_reviews`：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
reviewer_id INTEGER NOT NULL
reviewee_id INTEGER NOT NULL
project_id INTEGER
rating INTEGER NOT NULL
content TEXT
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
```

### 規則
- 評分 1～5 星。
- 留言可選。
- 評價顯示在個人頁面下方。
- 評價留言要有檢舉按鈕。
- 個人頁面本身不要放檢舉按鈕。

### 顯示
```txt
平均評分：4.5 / 5
共 8 則評價
```

## 功能 3：super_admin 設定管理員
只有 `role === 'super_admin'` 的使用者可以看到：

```txt
設為管理員
取消管理員
```

限制：
- 不能取消自己的 super_admin。
- 被設為管理員的人 role 改成 `admin`。
- 被取消管理員的人 role 改回 `user`。
- admin 不能設定別人為 admin。

## 階段 5 完成標準
- 個人頁面完整顯示資料。
- 使用者可以編輯自己的資料。
- 可以新增合作評價。
- 評價留言有檢舉按鈕。
- admin2006 可以設定 / 取消其他人的管理員身份。

---

# 階段 6：通知中心

## 目標
等前面功能完成後，再集中串接通知事件。

## 資料表
新增 `notifications`：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
user_id INTEGER NOT NULL
type TEXT NOT NULL
title TEXT NOT NULL
content TEXT
link TEXT
is_read INTEGER DEFAULT 0
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
```

## 右上角通知 UI
- 右上角顯示通知 icon。
- 有未讀通知時顯示紅點或數字。
- 點通知 icon 後，在右上角展開懸浮小視窗。
- 視窗固定寬度，建議 320px～380px。
- 高度過高時內部滾動，不影響主畫面。
- 點外面可關閉。
- 通知可以標記已讀。

## 通知事件
需要產生通知的事件：
- 有人申請加入專案。
- 申請被接受 / 拒絕。
- 被主動邀請加入專案。
- 邀請被接受 / 拒絕。
- 留言被回覆。
- 專案內容被修改。
- 被移出隊伍。
- 隊長轉移。
- 檢舉處理結果。
- 被設定為管理員。
- 管理員身份被取消。

## 階段 6 完成標準
- 右上角通知可打開懸浮視窗。
- 通知不影響主畫面排版。
- 前面功能的重要事件都有通知。
- 未讀狀態可以正確顯示。

---

# 階段 7：檢舉功能與管理員頁面

## 目標
最後做檢舉與管理員處理，因為它依賴使用者、留言、評價、通知。

## 功能 1：檢舉
### 資料表
新增 `reports`：

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
reporter_id INTEGER NOT NULL
target_type TEXT NOT NULL
target_id INTEGER NOT NULL
reason TEXT NOT NULL
status TEXT DEFAULT 'pending'
handler_id INTEGER
result_note TEXT
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
handled_at DATETIME
```

### target_type
```txt
project
comment
review
```

### 檢舉按鈕位置
需要：
- 專案
- 留言
- 評價留言

不需要：
- 個人頁面本身

### 檢舉流程
- 使用者點檢舉按鈕。
- 開啟檢舉懸浮視窗。
- 選擇或輸入檢舉原因。
- 送出後建立 report。
- 管理員收到通知。

---

## 功能 2：管理員頁面
管理員可以進入管理員頁面。

### 可進入者
```txt
role === 'admin'
role === 'super_admin'
```

### 管理員功能
- 查看會員列表。
- 停權 / 解除停權使用者。
- 查看專案列表。
- 刪除違規專案。
- 查看檢舉列表。
- 處理檢舉。
- 填寫處理結果。
- 通知檢舉人處理結果。

### super_admin 額外功能
只有 `admin2006`：
- 到使用者個人頁面設定對方為管理員。
- 取消對方管理員身份。

### 停權規則
被停權使用者：
- 不能建立專案。
- 不能申請加入專案。
- 不能留言。
- 不能評價。
- 可以登入，但看到停權提示。

## 階段 7 完成標準
- 使用者可以檢舉專案、留言、評價留言。
- 管理員可以處理檢舉。
- 管理員可以停權 / 解除停權。
- 管理員可以刪除違規專案。
- 檢舉處理結果會通知使用者。

---

# 階段 8：整合測試與 UI 修正

## 目標
最後檢查整體功能、權限、版面、RWD。

## 必測情境

### 一般使用者
- 註冊 / 登入
- 編輯個人資料
- 收藏專案
- 用已收藏篩選
- 申請加入專案
- 接受邀請
- 留言與回覆
- 評價合作對象
- 檢舉專案 / 留言 / 評價

### 隊長
- 建立專案
- 修改專案
- 新增公告
- 新增日期倒數
- 邀請成員
- 轉移隊長
- 滿員後不能邀請或接受申請

### admin

## 四、重要注意事項

1. 不要一次實作所有功能。
2. 每個階段都要先確認目前網站可以正常啟動。
3. 修改資料庫時要注意既有資料不要壞掉。
4. 所有新增 API 都要檢查登入狀態。
5. 所有需要權限的 API 都要檢查 role。
6. 所有懸浮視窗都要能關閉。
7. 手機版不能超出畫面。
8. 管理員 `[ADMIN]` 顯示請做成共用元件或共用函式，不要每個頁面各寫一次。
9. 公告和日期倒數只能在頁面上各顯示一行，完整內容必須點 div 後用懸浮視窗顯示。
10. 個人頁面本身不要檢舉按鈕，只有下方評價留言需要檢舉按鈕。
